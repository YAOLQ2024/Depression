# 量表流程与风险评估触发机制设计方案（V1）

更新时间：2026-04-02

文档文件名：`scale_screening_flow_and_risk_trigger_design_v1_20260402.md`

## 1. 文档定位

本文档用于回答以下几个在前端接入前必须先定清楚的问题：

1. `SDS（抑郁自评量表）`、`PHQ-9（患者健康问卷）`、`GAD-7（广泛性焦虑量表）` 在前端入口上应该如何组织
2. 三个量表的推荐答题顺序怎么安排
3. 用户是否必须答完三个量表
4. `risk-assessment（风险评估与安全建议模块）` 应该在什么时候触发
5. 如果用户只完成 `1` 个或 `2` 个量表，系统是否仍然允许生成初步风险评估
6. 前端页面和后端会话应该如何配合

本文档的目标不是细化具体视觉设计，而是先把产品流程和工程触发机制定下来，避免后面接前端和后端时逻辑反复变化。

## 2. 当前问题

当前系统前端存在两个并行入口：

1. `量表筛查`
当前位于 `/scales`，主要承载 `PHQ-9（患者健康问卷）` 和 `GAD-7（广泛性焦虑量表）`

2. `SDS 旧版测评`
当前在侧边栏单独占一项，承载旧链路中的多模态采集和完整测评

这会带来几个问题：

1. 三个量表都是量表，但入口被拆成了两套，用户理解成本高
2. `SDS（抑郁自评量表）` 被放在“旧版”路径下，不利于后续统一产品叙事
3. 如果直接把三个量表强行串成一个超长问卷，会显著增加放弃率
4. 如果每完成一个量表就立即调用 `risk-assessment（风险评估与安全建议模块）`，会造成不必要的重复计算

## 3. 总体结论

当前 `V1（第一版）` 推荐采用下面这套方案：

1. 前端侧边栏只保留一个主入口：`量表筛查`
2. `SDS 旧版测评` 不再作为长期一级导航，而是作为 `量表筛查` 页面中的一个量表卡片或临时兼容入口
3. 三个量表组成一条 `recommended screening flow（推荐筛查流程）`
4. 推荐流程顺序为：`PHQ-9 -> GAD-7 -> SDS`
5. 每个量表独立提交、独立保存，不把三个量表硬串成一个大表单
6. 不强制用户必须答完三个量表
7. `risk-assessment（风险评估与安全建议模块）` 不在每次量表提交后触发，而是在“本轮量表流程结束”时触发一次
8. 若用户只完成部分量表，系统仍允许生成 `初步风险评估`
9. 若用户完成三个量表并且 `SDS（抑郁自评量表）` 链路采到了多模态，则生成 `完整风险评估`

## 4. 前端入口设计

### 4.1 一级导航建议

侧边栏建议统一成：

1. `工作台`
2. `量表筛查`
3. `AI 咨询助手`
4. `历史记录`
5. `个人中心`
6. `其他数据管理与设置项`

说明：

1. 这里调整的是“测评相关入口如何收口”，不是删除整站已有主导航
2. `工作台` 和 `个人中心` 仍然属于长期保留的一级导航
3. 当前需要收口的是 `SDS（抑郁自评量表）` 不再长期单独占据一级导航

不建议继续保留：

1. `SDS 旧版测评`

原因：

1. 它会给用户造成“这不是统一量表体系的一部分”的错觉
2. 后续产品演进时，`SDS（抑郁自评量表）` 需要从“旧版孤岛”回到统一入口
3. 老师要求三个量表都纳入同一套系统流程

### 4.2 `/scales` 页面建议

`/scales` 页面建议升级为统一量表中心，包含两个区域：

1. `推荐筛查流程（推荐）`
2. `单项量表选择`

推荐筛查流程区域：

1. 展示一条三步流程
2. `PHQ-9（患者健康问卷）`
3. `GAD-7（广泛性焦虑量表）`
4. `SDS（抑郁自评量表，多模态完整评估）`

单项量表选择区域：

1. `PHQ-9 抑郁快速筛查`
2. `GAD-7 焦虑快速筛查`
3. `SDS 多模态完整评估`

这样设计的好处：

1. 既体现老师要求的“三个量表都要纳入系统”
2. 又保留用户自由度
3. 方便后续逐步把 `SDS` 从旧页面迁到新页面，而不是一次性硬切

## 5. 推荐答题顺序

### 5.1 推荐顺序

当前建议：

1. `PHQ-9（患者健康问卷）`
2. `GAD-7（广泛性焦虑量表）`
3. `SDS（抑郁自评量表）`

### 5.2 这样安排的原因

1. `PHQ-9（患者健康问卷）` 题量较少，且能快速给出抑郁风险概貌与 `第 9 题安全信号`
2. `GAD-7（广泛性焦虑量表）` 题量也较少，能补足焦虑维度
3. `SDS（抑郁自评量表）` 题量更多，而且当前与多模态采集链路绑定最深，放在最后更适合承接完整评估

### 5.3 不建议的方案

不建议：

1. 让用户一上来先做 `SDS（抑郁自评量表）`
2. 把三个量表题目完全串成一个超长大表单

原因：

1. `SDS（抑郁自评量表）` 前置会抬高完成门槛
2. 三个量表全串会显著拉长单次作答时长
3. 对社区场景和普通用户来说，过长流程会明显增加退出率

## 6. 是否强制答完三个量表

### 6.1 结论

当前建议：

1. `不强制`
2. `强推荐`

也就是说：

1. 系统要明确推荐用户完成三量表流程
2. 但不应该强制要求用户必须全部答完才能继续

### 6.2 原因

如果强制三个量表都做完，问题会很明显：

1. 完成率下降
2. `SDS（抑郁自评量表）` 题量和多模态采集门槛更高
3. 用户容易在中途退出，最终反而拿不到任何有效结果

如果完全放任用户自由选择，又会出现：

1. 数据完整度差异大
2. 不同人只做一个或两个量表时，结果口径不一致

所以最合理的平衡是：

1. 三量表流程作为主推荐路径
2. 用户可以在任意一步结束
3. 但前提是至少已经完成 `1` 份完整量表
4. 系统基于当前已完成的完整量表给出 `初步风险评估`
5. 只有当三量表和多模态都比较齐全时，再给出 `完整风险评估`

## 7. 单量表提交与风险评估触发的关系

### 7.1 核心原则

不建议每完成一个量表就立刻调用 `risk-assessment（风险评估与安全建议模块）`。

正确做法是：

1. 每个量表提交时，只做该量表自身计分和保存
2. 只有在“本轮量表流程结束”时，才调用一次 `risk-assessment`
3. 未完成的量表草稿不参与 `risk-assessment`

### 7.2 为什么不能每次都调

如果每次提交都调：

1. 做完 `PHQ-9（患者健康问卷）` 调一次
2. 做完 `GAD-7（广泛性焦虑量表）` 再调一次
3. 做完 `SDS（抑郁自评量表）` 再调一次

这样会带来：

1. 重复计算
2. 无意义的资源占用
3. 接口调用频率变高
4. 后面加入多模态和 `LLM（大语言模型）` 后会更不划算

因此当前推荐：

1. 单量表提交不触发 `risk-assessment`
2. 流程结束时触发一次 `risk-assessment`
3. 只有“至少完成 1 份完整量表”时才允许触发
4. 未完成量表只支持保存进度，不生成风险评估

### 7.3 草稿暂存策略

当前 `V1（第一版）` 已落地的草稿策略如下：

1. `SDS（抑郁自评量表）` 采用服务端草稿暂存
2. `PHQ-9（患者健康问卷）` 与 `GAD-7（广泛性焦虑量表）` 也采用服务端草稿暂存
3. 草稿不只保存在浏览器本地，而是持久化到后端数据库
4. 页面再次打开时，会优先读取服务端草稿并恢复已选答案、已用时间和最近作答位置
5. 量表正式提交成功后，对应草稿会被标记完成，不再作为可恢复草稿返回

这样做的工程意义是：

1. 用户刷新页面、切换设备或重新登录后仍可恢复进度
2. 草稿和 `screening-flow-session（量表流程会话）` 可以保持一致
3. 后续如果要迁移到 `Redis（内存缓存）` 或独立草稿服务，前端交互不需要重写

## 8. 风险评估的触发时机

### 8.1 允许触发的三个时机

当前建议只在以下时机触发：

1. 用户完成推荐流程中的最后一个量表后
2. 用户已经完成至少 `1` 份完整量表，并在中途主动选择结束本轮流程时
3. 用户已经完成至少 `1` 份完整量表，准备离开流程时，系统询问是否基于当前已完成量表生成风险评估

### 8.2 具体触发规则

规则一：完成最后一个量表

1. 如果用户按推荐顺序完成到了 `SDS（抑郁自评量表）`
2. 则系统自动进入 `风险评估准备态`
3. 再自动或半自动触发 `risk-assessment`

规则二：中途主动结束

1. 如果用户完成 `PHQ-9（患者健康问卷）`
2. 或完成 `PHQ-9 + GAD-7`
3. 但不想继续做后续量表
4. 则用户可以点击：
5. `结束本轮并生成当前风险评估`

规则三：中途退出确认

1. 如果用户尝试离开流程页
2. 且当前已经至少完成 `1` 份完整量表
3. 系统可弹出确认框：
4. `是否基于当前已完成量表生成初步风险评估？`

规则四：未完成当前量表时

1. 如果用户当前正在填写某个量表，但该量表尚未完整作答
2. 则该量表不参与 `risk-assessment（风险评估与安全建议模块）`
3. 系统只允许：
4. `save draft（保存草稿）`
5. `resume later（下次继续）`
6. 此时页面应提示：
7. `当前量表尚未完成，暂时无法生成风险评估`

## 9. 页面按钮与文案建议

### 9.1 每个量表结果页的主按钮

当前建议每个量表结果页都提供两个主按钮：

1. `继续下一量表（推荐）`
2. `结束本轮并生成当前风险评估`

如果已经是最后一个量表，则按钮改为：

1. `生成完整风险评估`

说明：

1. 上述按钮只出现在“某份量表已经完整提交并出结果”之后
2. 不应在“量表尚未答完”的页面直接展示 `结束本轮并生成当前风险评估`

### 9.1.1 未完成量表页面按钮

当前建议：

1. `保存进度`
2. `继续作答`
3. `返回量表中心`

不建议出现：

1. `基于当前未完成量表生成风险评估`

### 9.2 页面提示语

建议文案：

1. `当前结果仅代表本量表筛查结果，不等于完整风险评估。`
2. `继续完成后续量表，可获得更完整的初步风险判断。`
3. `如不想继续，也可以基于当前已完成量表生成初步风险评估。`
4. `当前量表尚未完成时，系统仅保存进度，不生成风险评估。`

### 9.3 风险评估结果页标题建议

根据完成度区分：

1. `初步风险评估（部分量表）`
2. `完整风险评估（三量表 + 可用多模态）`

## 10. 后端流程会话设计

### 10.1 为什么需要会话

如果三个量表是分步独立提交的，那么系统必须知道：

1. 当前这一轮流程里，用户已经做了哪些量表
2. 哪些量表结果属于同一轮流程
3. 用户是主动结束，还是流程自然完成
4. 哪些量表只是草稿，哪些量表已经完整完成

因此建议引入：

`screening-flow-session（量表流程会话）`

### 10.2 建议字段

建议至少包含：

```json
{
  "flow_session_id": "flow-20260402-001",
  "user_id": "user-1001",
  "status": "in_progress",
  "recommended_order": ["phq9", "gad7", "sds"],
  "completed_scales": ["phq9", "gad7"],
  "draft_scales": ["sds"],
  "phq9_record_id": 101,
  "gad7_record_id": 102,
  "sds_record_id": null,
  "has_speech": false,
  "has_emotion": false,
  "has_eeg": false,
  "risk_assessment_generated": false,
  "started_at": "2026-04-02T16:00:00+08:00",
  "updated_at": "2026-04-02T16:12:00+08:00"
}
```

### 10.3 状态建议

建议至少定义：

1. `in_progress（进行中）`
2. `ready_for_risk_assessment（可生成风险评估）`
3. `risk_assessment_completed（风险评估已生成）`
4. `abandoned（中途放弃）`

补充原则：

1. 只有 `completed_scales（已完成量表）` 里的量表可以参与 `risk-assessment`
2. `draft_scales（草稿量表）` 只用于恢复进度，不参与评分与风险判断

## 11. `risk-assessment` 的输入调整建议

### 11.1 当前问题

当前 `risk-assessment（风险评估与安全建议模块）` 更偏向接收单个：

1. `assessment_summary（量表摘要）`

但按本方案，后续更合理的输入应该是：

1. `assessment_bundle（量表结果集合）`

### 11.2 建议升级方向

建议后续逐步升级为：

```json
{
  "flow_session_id": "flow-20260402-001",
  "assessment_bundle": {
    "phq9_summary": {},
    "gad7_summary": {},
    "sds_summary": {}
  },
  "speech_summary": {},
  "emotion_summary": {},
  "eeg_summary": {},
  "completion_state": {
    "completed_scales": ["phq9", "gad7"],
    "is_partial": true,
    "is_recommended_flow_complete": false
  }
}
```

### 11.3 部分量表的处理原则

如果用户只完成了部分量表：

1. 允许评估
2. 但结果要明确标注为 `初步风险评估`
3. `assessment_confidence（评估置信度）` 应低于完整量表场景
4. 页面和报告中应提示：
5. `当前结果基于部分量表生成，建议完成后续量表以提高完整性`

补充说明：

1. 这里的“部分量表”指的是“完整做完了 1 份或 2 份量表，但没有完成全部推荐流程”
2. 不包括“某一份量表只答了一部分题目”的情况

## 12. 当前推荐的最终方案

### 12.1 产品层结论

1. `SDS（抑郁自评量表）` 不再长期单独占据一级导航
2. 三个量表统一进入 `/scales`
3. 推荐流程为 `PHQ-9 -> GAD-7 -> SDS`
4. 每个量表独立提交，不做超长总表单
5. 用户不被强制要求答完三个量表
6. 系统支持“部分量表初步评估”和“完整量表完整评估”
7. 未完成的量表只保存草稿，不参与风险评估

### 12.2 工程层结论

1. 单量表提交只负责该量表计分和保存
2. `risk-assessment（风险评估与安全建议模块）` 不在每次量表提交时触发
3. 仅在“至少完成 1 份完整量表且本轮流程结束”时触发一次
4. 未完成量表只进入草稿，不参与风险评估
5. 需要引入 `screening-flow-session（量表流程会话）`
6. 后续 `risk-assessment` 输入建议从单个 `assessment_summary` 升级到 `assessment_bundle`

## 13. 实施优先级建议

当前推荐按下面顺序推进：

1. 先改 `/scales` 页面文案和入口组织
2. 再确定 `PHQ-9 -> GAD-7 -> SDS` 的页面跳转逻辑
3. 再补 `结束本轮并生成当前风险评估` 的按钮和触发动作
4. 再设计 `screening-flow-session（量表流程会话）`
5. 最后再把 `risk-assessment` 从单量表输入升级成量表集合输入

## 14. 当前不建议立刻做的事

当前不建议：

1. 立刻把 `SDS（抑郁自评量表）` 的旧前端完全重写
2. 立刻把三个量表完全并成一个页面
3. 立刻要求 `risk-assessment` 必须支持所有量表组合情况的复杂融合

更稳的做法是：

1. 先定产品流
2. 再定会话结构
3. 再逐步接入与演进
